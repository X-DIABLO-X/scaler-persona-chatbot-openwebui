# Reflection

This project reinforced the GIGO principle in a very practical way. The biggest lesson was that persona quality depends much less on the frontend and much more on the structure of the system prompt. A generic prompt can produce technically correct answers, but it will not feel like a specific person. The moment I made the prompts more explicit about tone, constraints, and example responses, the chatbot became noticeably more believable. The few-shot examples were especially important because they anchored each persona to a repeatable style instead of relying on the base model to guess what "sound like this person" means.

Another thing that worked well was treating persona selection as model selection inside Open WebUI. Instead of building a custom frontend from scratch, I used Open WebUI as the chat layer and placed a small OpenAI-compatible proxy in front of Groq. That proxy exposed exactly three models, each mapped to a different system prompt. This made the user experience clean and aligned with the assignment requirement that the active persona should always be visible. It also kept the backend logic centralized, which made prompt management simpler.

Using a separate prompt-guard model before the main LLM was also valuable. It added a lightweight safety layer and helped prevent prompt injection attempts from overriding persona instructions. This reminded me that good prompt engineering is not only about writing better prompts, but also about protecting them from being bypassed.

If I had more time, I would improve two things. First, I would do deeper public research on each persona by reviewing more interviews, talks, and posts so that the prompts could reflect more distinctive phrases, beliefs, and examples. Second, I would add a thin customization layer on top of Open WebUI to reset the conversation more visibly when the persona changes, making that transition even more explicit for the evaluator. Overall, the project showed that prompt engineering is closer to product design than simple text writing: the quality of the input specification directly shapes the quality, consistency, and trustworthiness of the output.
